# Issue Quality

Good contributor issues reduce ambiguity. They tell contributors what problem exists, what outcome is expected, which files are likely involved, how maintainers will review the work, and whether Stellar or Soroban context matters.

## What makes a good contributor issue

- A problem statement that describes a real user, maintainer, or developer pain.
- Expected behavior that describes the finished state.
- Files or folders that help contributors orient quickly.
- Acceptance criteria that can be checked during review.
- Difficulty level for self-selection.
- Labels for type, domain, triage, and contributor fit.

## Why acceptance criteria matter

Acceptance criteria turn an idea into a reviewable contract. They reduce back-and-forth, help contributors test their work, and give maintainers a clear reason to approve or request changes.

## Label recommendations

Use type labels such as `bug`, `enhancement`, `documentation`, and `testing`. Add domain labels such as `frontend`, `backend`, `wallet`, `stellar`, and `soroban`. Add workflow labels such as `good first issue`, `help wanted`, `maintainer-approved`, and `needs triage`.

## Difficulty levels

`beginner` issues should be small, well documented, and low risk. `intermediate` issues can touch multiple files or require domain familiarity. `advanced` issues may involve architecture, cross-cutting behavior, or deeper Stellar/Soroban knowledge.

## Stellar/Soroban relevance

Explain whether the task touches wallets, RPC configuration, contracts, transaction status, network selection, account behavior, or ecosystem documentation. If it does not, say so plainly.
